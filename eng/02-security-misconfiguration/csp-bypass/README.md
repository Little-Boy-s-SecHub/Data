# Content Security Policy (CSP) Bypass

> [!CAUTION]
> Only practice on systems you own or are explicitly authorized. Use synthetic data, disposable fixtures, and limit resources; do not send payloads to the Internet or real targets.

## 1. Learning Objectives

After this lesson, you can:

- Explain Content Security Policy (CSP) Bypass using the root cause instead of just describing the impact. 
- Identify trust boundaries, assets, actors, and the necessary conditions for the vulnerability to be exploited. 
- Conduct controlled testing in a local lab and differentiate expected results from false positives. 
- Select the root control, implement the fix, and retest using positive, negative, and boundary cases.

## 2. Prerequisites

- CSP Level 3 directive resolution and source expression.

- Inline script, nonce/hash, `base-uri` and DOM sink.

- Chromium fixture has a local console/report collector.

## 3. Background Knowledge

Imagine your website as a castle throwing a party. You hire a strict butler and give them a list that says: "Tonight, we only accept wine from our own cellar (`'self'`) and a trusted shop, which is `cdn.trusted.lab.test`. Anyone bringing food or drinks from other sources must be stopped at the door." This list of safety rules is called **CSP** (Content Security Policy). It is an extremely strong shield sent from the server via the HTTP header to instruct the browser: "Only scripts, images, or interfaces from sources on this safe list are allowed to run." [S7]

CSP is a defense-in-depth layer that can make many payloads **XSS** (Cross-Site Scripting) harder to exploit, but it does not fix the sink XSS and does not provide absolute protection against all contexts/browsers/policies. With nonce-based CSP, the browser only allows inline scripts with a nonce that matches the policy; the nonce must be new for each response, hard to guess, and not attached to content controlled by the attacker. The application still needs to use proper output encoding/sanitization for the correct context and API DOM safely. [S6]

Each directive in CSP will manage a separate type of resource: `script-src` manages JavaScript code, `style-src` manages the CSS interface, and `default-src` is the final checkpoint for everything else. The browser will read and analyze these directives to enforce strict protection rules on the page. [S7]

In a response using a nonce, the same value must appear in the script-source directive of the header and on the trusted script element generated by the server. The value must be new, hard to guess for each response, and must not be attached to user-controlled content. Section 8 contains two response fixtures that have been annotated to observe cases of missing nonce and reused nonce; section 9 contains secure nonce generation configuration. [S6] [S7]

## 4. Description and Root Cause

The **Bypass CSP** vulnerability (Bypassing content security policy) does not occur because the CSP technology itself is flawed, but rather arises from developers setting protection rules too carelessly or too loosely. It's like a housekeeper holding a permission list that says: "Accept food from any sources outside the street (`unsafe-inline`)" or "Allow guests to freely change the delivery address." This is not a flaw of the CSP specification but a configuration error — a typical form of Security Misconfiguration. [S7]

Common mistakes include allowing direct execution of code without verification (`unsafe-inline`), allowing the execution of character strings as code (`unsafe-eval`), lacking `base-uri` directives, or placing large domains containing public services (such as shared CDN or JSONP data reception points) on the trusted list. [S7]


## 5. Threat Model and Exploitation Conditions

- **Assets:** the right to run scripts/change the base URL in the lab document; marker only located in DOM.

- **Actor:** the actor has the ability to insert inline scripts or markup according to each scenario.

- **Trust boundary:** The CSP Level 3 response header is applied by Chromium to HTML from Express.js.

- **Necessary condition:** directive allows correct sink, for example unsafe-inline or missing base-uri; nonce/hash must be considered separately.

- **Environment:** Chromium pinned version, page 127.0.0.1:9000, asset mock 9001, no outbound.

Weak configuration is a prerequisite; bypass evidence must be an execution marker or a request asset with the correct context and policy. [S6]

## 6. Attack Mechanism

The browser applies each CSP directive to the corresponding sink. unsafe-inline allows inline markers in the specified context; missing base-uri only has an effect when markup injection changes the resolution of relative resources. [S6]

## 7. Testing in an Authorized Lab

1. **Setup:** run the page/asset fixture loopback, pin Chromium and enable CSP report/DevTools log.
2. **Baseline:** policy nonce/hash or base-uri 'none' blocks the corresponding payload.
3. **Action:** apply each header/HTML pair as annotated, do not mix unsafe-inline and base-uri in one case.
4. **Expected result:** weak policy generates DOM/request marker; fixed policy blocks and generates a violation log.
5. **Boundary:** check CSP2/CSP3, multiple headers, report-only and cache.
6. **Cleanup:** delete browser profile, marker and report; stop fixture.

**Case 1: `unsafe-inline` weakens `script-src`**

`'unsafe-inline'` allows inline script and inline event handler when the policy has no nonce/hash handled by CSP3. It does not “disable all of CSP”: other directives still take effect, and modern browsers ignore `'unsafe-inline'` when there is a valid nonce/hash. [S6]

## 9. Vulnerable Code and Secure Code

```javascript
// === VULNERABLE CSP (Express.js) ===
app.use((req, res, next) => {
    // BAD: unsafe-inline + unsafe-eval + wildcard subdomain
    res.setHeader('Content-Security-Policy',
        "default-src 'self'; " +
        "script-src 'self' 'unsafe-inline' 'unsafe-eval' https://*.cdn.lab.test; " +
        "style-src 'self' 'unsafe-inline'"
    );
    next();
});


// === SECURE CSP (Express.js with nonce) ===
const crypto = require('crypto');

app.use((req, res, next) => {
    // Generate cryptographically random nonce per request
    const nonce = crypto.randomBytes(16).toString('base64');
    res.locals.cspNonce = nonce;

    // GOOD: Strict nonce-based CSP with all critical directives
    res.setHeader('Content-Security-Policy', [
        "default-src 'none'",
        `script-src 'nonce-${nonce}' 'strict-dynamic'`,
        "style-src 'self'",
        "img-src 'self' data:",
        "connect-src 'self'",
        "font-src 'self'",
        "base-uri 'self'",
        "form-action 'self'",
        "frame-ancestors 'none'",
        "report-uri /csp-report" // Retain for compatibility while adopting Reporting API
    ].join('; '));
    next();
});

// Render the same per-response value into trusted script elements with the
// template engine's HTML-attribute escaping enabled.
// EJS example: <script nonce="<%= cspNonce %>">...</script>
```

## 10. Detection

- Load the fixture using Chromium and observe the script marker, console violation, and CSP report. [S7]

- Review effective directive, nonce lifecycle, host source, and fallback directives. [S7]

- Compare actual headers on success/error/redirect responses, not just the template configuration.

## 11. Defense

### Compulsory control

- Use a narrow policy according to the resource type; create a new hard-to-guess nonce for each response and attach it only to trusted scripts. [S7]

- Limit `base-uri`, objects, and host scripts according to actual needs. [S7]

### Defense-in-depth

- CSP supplements output encoding and safe DOM API; does not fix the root cause XSS.

- Test the report in trial mode before enforcing.

## 12. Retest

- **Positive:** the script with a valid nonce runs and the necessary resources are loaded.

- **Negative:** inline script without nonce and external base outside the policy is blocked.

- **Boundary:** error page, cached HTML, nonce reuse, and directive fallback.

- **Telemetry:** check the console, violation report, and DOM marker.

## 13. Common Mistakes

- Reusing a nonce or allowing the actor to insert it into a tag with a nonce.

- Allowlist a host containing JSONP/script controlled by someone else.

- Forget `base-uri` when using relative script URL.

- Calling a policy 'bypass' just because it is missing a directive and does not yet have an execution path.

## 15. Glossary

- **CSP:** browser policy controlling the source and the conditions for loading/executing resources. [S7]

- **Nonce:** an unpredictable value, new for each response, used to allow specific scripts/styles. [S7]

- **Effective directive:** the directive that actually governs the type of resource after the fallback rules of CSP. [S7]
